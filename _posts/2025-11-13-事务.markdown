## 1.分布式事务

### 2PC（两阶段提交）

核心思想：引入一个**协调者**（Transaction Manager）来统一控制所有参与者的提交/回滚。

**第一阶段（Prepare）**：协调者询问所有参与者是否可以提交，参与者执行事务操作但不提交，返回 Yes/No

**第二阶段（Commit/Rollback）**：如果所有参与者都返回 Yes，协调者发出 Commit；否则发出 Rollback

**缺点**：
- 同步阻塞：Prepare 阶段所有参与者锁定资源等待，性能差
- 单点故障：协调者宕机，参与者会一直阻塞
- 数据不一致：第二阶段部分 Commit 成功部分失败

### 3PC（三阶段提交）

在 2PC 基础上增加了 **CanCommit** 阶段和**超时机制**，降低了阻塞风险。

**第一阶段（CanCommit）**：轻量级检查，参与者判断自身是否有能力完成事务（不锁资源）

**第二阶段（PreCommit）**：类似 2PC 的 Prepare，执行事务但不提交

**第三阶段（DoCommit）**：真正提交，并引入超时自动提交机制

**优势**：参与者引入超时机制，避免永久阻塞；但仍无法完全解决数据不一致问题

### TCC（Try-Confirm-Cancel）

业务层面的分布式事务，需要**业务侥入**，每个服务都要实现 Try、Confirm、Cancel 三个接口。

| 阶段 | 职责 | 示例（转账） |
|------|------|------|
| Try | 资源预留/检查 | 冻结转账金额，但不真正扣款 |
| Confirm | 确认执行 | 真正扣款并到账 |
| Cancel | 取消预留 | 解冻金额，恢复原状 |

**优点**：不锁数据库资源，性能高；最终一致性

**缺点**：业务侥入性强，开发成本高；需考虑幂等、空回滚、悬挂等问题

### XA

数据库层面的 2PC 实现，由数据库原生支持。

- 事务管理器（TM）协调多个资源管理器（RM，即数据库）
- 数据库原生支持 XA 协议（MySQL、Oracle、PostgreSQL 等）
- **强一致性**，但性能差（长时间锁定资源）
- 适用场景：对一致性要求极高、并发量不大的场景

### Saga

长事务解决方案，将一个大事务拆分为多个本地小事务，通过**补偿机制**保证最终一致性。

```
T1 → T2 → T3 → T4（失败）
                    ↓
          C3 ← C2 ← C1（反向补偿）
```

**两种模式**：
- 编排式（Orchestration）：中心协调器控制流程（Seata Saga 采用）
- 协同式（Choreography）：各服务通过事件驱动，无中心控制

**优点**：无全局锁，高性能；适合长流程业务

**缺点**：不保证隔离性；补偿逻辑开发复杂

### Seata 框架

Seata 是阿里开源的分布式事务解决方案，支持上述多种模式：

| 模式 | 一致性 | 性能 | 业务侥入 | 适用场景 |
|------|--------|------|----------|----------|
| AT | 最终一致 | 高 | 无 | 常规业务（推荐默认） |
| TCC | 最终一致 | 最高 | 强 | 资金、库存等高性能场景 |
| XA | 强一致 | 低 | 无 | 对一致性要求极高场景 |
| Saga | 最终一致 | 高 | 中 | 长流程业务 |

**Seata AT 模式原理**（最常用）：
- 一阶段：业务 SQL 和回滚日志（undo_log）在同一个本地事务中提交，释放本地锁和连接资源
- 二阶段（提交）：异步删除 undo_log
- 二阶段（回滚）：根据 undo_log 反向补偿

### 分布式事务事件监听

```java
public enum TransactionPhase {
    // 事务即将提交但尚未提交
    BEFORE_COMMIT,
    // 事务已提交
    AFTER_COMMIT,
    // 事务已回滚
    AFTER_ROLLBACK,
    // 事务完成（无论提交还是回滚）
    AFTER_COMPLETION;
}
```

通过 `@TransactionalEventListener` 注解监听事务事件，常用于事务提交后发送消息、清理缓存等场景：

```java
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void handleOrderCreated(OrderCreatedEvent event) {
    // 事务提交后才执行，避免事务回滚但消息已发送的问题
    messageSender.send(event);
}
```

## 2.Spring事务 - 核心接口

```java
// 事务管理的核心入口
public interface PlatformTransactionManager {
    TransactionStatus getTransaction(TransactionDefinition definition) throws TransactionException;
    void commit(TransactionStatus status) throws TransactionException;
    void rollback(TransactionStatus status) throws TransactionException;
}

// 事务定义（传播行为、隔离级别、超时等）
public interface TransactionDefinition {
    int getPropagationBehavior();
    int getIsolationLevel();
    int getTimeout();
    boolean isReadOnly();
    String getName();
}

// 事务状态
public interface TransactionStatus {
    boolean isNewTransaction();
    boolean hasSavepoint();
    void setRollbackOnly();
    boolean isRollbackOnly();
    boolean isCompleted();
}
```

## 4.Spring事务 - 传播行为

传播行为定义了当一个事务方法被另一个事务方法调用时，事务如何传播。

| 传播行为 | 说明 | 场景 |
|----------|------|------|
| **REQUIRED**（默认） | 当前有事务则加入，没有则新建 | 大多数业务方法 |
| **REQUIRES_NEW** | 总是新建事务，挂起当前事务 | 日志记录、审计（不能因主事务回滚而丢失） |
| **NESTED** | 当前有事务则创建嵌套事务（savepoint） | 部分失败可回滚子操作，不影响主事务 |
| **SUPPORTS** | 有事务就用，没事务就不用 | 查询方法 |
| **NOT_SUPPORTED** | 以非事务方式执行，挂起当前事务 | 不需要事务的操作 |
| **MANDATORY** | 当前必须存在事务，否则抛异常 | 强制要求调用方开启事务 |
| **NEVER** | 以非事务执行，当前存在事务则抛异常 | 禁止在事务中调用 |

**REQUIRED vs REQUIRES_NEW vs NESTED 对比**：

```java
// REQUIRED：A和B在同一个事务中，B异常会导致A也回滚
@Transactional(propagation = Propagation.REQUIRED)
public void methodA() {
    methodB();  // B抛异常 → A也回滚
}

// REQUIRES_NEW：B独立事务，B回滚不影响A，A回滚也不影响B
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void methodB() {
    // 独立事务，与外层无关
}

// NESTED：B是嵌套事务，B回滚不影响A，但A回滚会导致B也回滚
@Transactional(propagation = Propagation.NESTED)
public void methodB() {
    // savepoint，外层可捕获异常后继续
}
```

## 5.Spring事务 - 隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 说明 |
|----------|------|----------|------|------|
| READ_UNCOMMITTED | ✔ | ✔ | ✔ | 最低隔离，几乎不用 |
| READ_COMMITTED | ✘ | ✔ | ✔ | 默认（Oracle、PostgreSQL） |
| REPEATABLE_READ | ✘ | ✘ | ✔ | 默认（MySQL InnoDB） |
| SERIALIZABLE | ✘ | ✘ | ✘ | 最高隔离，性能最差 |

> MySQL InnoDB 在 REPEATABLE_READ 级别通过 MVCC + Next-Key Lock 已经解决了幻读问题。

## 6.@Transactional 工作原理

代理机制：Spring AOP 创建代理对象拦截方法调用

事务拦截：TransactionInterceptor 负责事务的开启、提交、回滚

事务管理：PlatformTransactionManager 抽象不同数据源的事务操作

资源绑定：通过 ThreadLocal 管理连接等事务资源

传播控制：根据传播行为决定事务的创建、加入、挂起等

同步回调：TransactionSynchronization 提供事务生命周期钩子

```bash
    /*
    事务执行时间线：
    
    T0: TransactionInterceptor.invoke() 开始
        ↓
    T1: 获取 @Transactional 配置
        ↓  
    T2: 根据传播行为处理现有事务
        ↓
    T3: DataSourceTransactionManager.doBegin()
        - 获取数据库连接
        - 设置隔离级别、只读等
        - 关闭自动提交
        - 绑定连接到ThreadLocal
        ↓
    T4: 执行目标业务方法
        - userRepository.save(user)
        - 其他数据库操作
        ↓
    T5: 业务方法执行完成
        ↓
    T6: 提交事务 DataSourceTransactionManager.doCommit()
        ↓
    T7: 清理ThreadLocal资源
        ↓
    T8: 返回结果
    
    异常情况：
    T4: 业务方法抛出异常
        ↓
    T5: 回滚事务 DataSourceTransactionManager.doRollback()
        ↓  
    T6: 清理ThreadLocal资源
        ↓
    T7: 重新抛出异常
    */
```

## 7.@Transactional 失效场景（面试高频）

| 失效场景 | 原因 | 解决方案 |
|----------|------|----------|
| **自调用**（this.method()） | 绕过了AOP代理，直接调用目标对象方法 | 注入自身代理 / AopContext / 拆分到另一个Bean |
| **方法非 public** | Spring AOP 默认只代理 public 方法 | 改为 public |
| **异常被吃掉** | catch了异常没抛出，Spring不知道要回滚 | 抛出异常 或手动 setRollbackOnly() |
| **异常类型不匹配** | 默认只回滚 RuntimeException/Error | 指定 rollbackFor = Exception.class |
| **未被 Spring 管理** | 类没有注册为 Bean | 加 @Service/@Component 等注解 |
| **多线程调用** | 事务绑定在 ThreadLocal，新线程无法共享 | 在同一线程内完成事务操作 |
| **数据库不支持** | 如 MyISAM 引擎不支持事务 | 使用 InnoDB |

**自调用失效详解**：

```java
@Service
public class OrderService {
    
    // ✘ 事务失效！this.bindCoupon() 直接调用目标对象，绕过代理
    @Transactional
    public void createOrder() {
        saveOrder();
        this.bindCoupon();  // 自调用，事务不生效
    }
    
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void bindCoupon() { ... }
    
    // ✔ 解决方案：注入自身代理
    @Autowired
    private OrderService self;  // Spring会注入代理对象
    
    @Transactional
    public void createOrderFixed() {
        saveOrder();
        self.bindCoupon();  // 通过代理调用，事务生效
    }
}
```

## 8.Spring事务 - rollbackFor 配置

```java
// 默认：只回滚 RuntimeException 和 Error
@Transactional
public void method() { ... }

// 推荐：显式指定回滚所有异常
@Transactional(rollbackFor = Exception.class)
public void method() { ... }

// 指定不回滚的异常
@Transactional(rollbackFor = Exception.class, noRollbackFor = BusinessException.class)
public void method() { ... }
```

## 9.本地事务表

**核心思想**：将分布式事务拆解为本地事务 + 异步消息，通过本地消息表记录事务状态，配合定时任务补偿，达到最终一致性。

**实现流程**：

1. **业务操作 + 写消息表**：在同一个本地事务中，执行业务操作并向本地消息表插入一条消息记录（状态为"待发送"）
2. **发送消息**：事务提交后，通过 MQ 将消息发送给下游服务
3. **下游消费确认**：下游服务消费成功后，回调通知上游将消息状态更新为"已完成"
4. **定时补偿**：定时任务扫描本地消息表中状态为"待发送"且超过一定时间的消息，重新发送
5. **兜底机制**：超过最大重试次数后，告警并人工介入处理

**本地消息表结构示例**：

```sql
CREATE TABLE local_message (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    biz_id VARCHAR(64) NOT NULL COMMENT '业务ID，用于幂等',
    topic VARCHAR(128) NOT NULL COMMENT '消息主题',
    body TEXT NOT NULL COMMENT '消息体（JSON）',
    status TINYINT NOT NULL DEFAULT 0 COMMENT '0-待发送 1-发送中 2-已完成 3-失败',
    retry_count INT NOT NULL DEFAULT 0 COMMENT '已重试次数',
    max_retry INT NOT NULL DEFAULT 5 COMMENT '最大重试次数',
    next_retry_time DATETIME COMMENT '下次重试时间',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_status_retry (status, next_retry_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**关键设计要点**：

- **原子性保证**：业务操作和消息入表在同一本地事务中，要么都成功，要么都回滚
- **幂等消费**：下游服务必须实现幂等，因为消息可能重复投递
- **退避策略**：重试间隔建议递增（如 1min、5min、15min、30min、1h），避免频繁重试造成压力
- **消息清理**：已完成的消息定期归档或删除，防止表膨胀

**适用场景**：跨服务的订单创建、积分发放、通知推送等对一致性要求为"最终一致"的场景

**优缺点**：
- 优点：实现简单，不依赖外部中间件的事务特性，可靠性高
- 缺点：与业务耦合较大，需要维护消息表，存在一定延迟

## 10.RocketMQ 事务消息

**核心思想**：RocketMQ 事务消息通过**半消息（Half Message）+ 事务回查**机制，确保本地事务执行与消息发送的原子性。本质上是一种二阶段提交的变体。

> 注意：RocketMQ 事务消息解决的是"消息一定能发出去"的问题（生产端原子性），下游消费仍需自行保证幂等和最终一致。

**执行流程**：

1. **发送半消息（Half Message）**：Producer 先向 Broker 发送半消息，此时消息对 Consumer 不可见
2. **执行本地事务**：Producer 执行本地业务逻辑（如扣款、下单等）
3. **提交/回滚**：
   - 本地事务成功 → 向 Broker 发送 Commit，消息变为可消费状态
   - 本地事务失败 → 向 Broker 发送 Rollback，消息被丢弃
4. **事务回查**：若 Broker 长时间未收到二次确认（如 Producer 宕机），Broker 会主动回查 Producer 的事务状态，Producer 根据本地事务结果返回 Commit 或 Rollback

**代码示例**：

```java
// 1. 创建事务消息生产者
TransactionMQProducer producer = new TransactionMQProducer("tx_producer_group");
producer.setNamesrvAddr("localhost:9876");

// 2. 设置事务监听器
producer.setTransactionListener(new TransactionListener() {
    
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        // 执行本地事务（如数据库操作）
        try {
            orderService.createOrder(arg);
            return LocalTransactionState.COMMIT_MESSAGE;
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }
    
    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 事务回查：根据业务ID查询本地事务状态
        String orderId = msg.getKeys();
        boolean exists = orderService.existsOrder(orderId);
        return exists ? LocalTransactionState.COMMIT_MESSAGE 
                      : LocalTransactionState.UNKNOW;  // UNKNOW会继续回查
    }
});

// 3. 发送事务消息
Message msg = new Message("order_topic", "创建订单消息体".getBytes());
msg.setKeys("ORDER_202311130001");
producer.sendMessageInTransaction(msg, orderParam);
```

**事务回查机制细节**：

- 默认回查次数：15 次（`transactionCheckMax`）
- 默认回查间隔：60 秒（`transactionCheckInterval`）
- 超过最大回查次数仍为 UNKNOW，消息将被丢弃并打印错误日志
- 回查时 Producer 需要根据业务数据（如订单表记录）判断事务是否成功

**关键设计要点**：

- **半消息存储**：RocketMQ 将半消息存储在内部 Topic（`RMQ_SYS_TRANS_HALF_TOPIC`），对消费者不可见
- **Op消息**：Commit/Rollback 后会写入 Op Topic（`RMQ_SYS_TRANS_OP_HALF_TOPIC`），标记已处理
- **回查线程**：Broker 端有 `TransactionalMessageCheckService` 线程定期扫描半消息，发起回查
- **幂等要求**：下游 Consumer 必须做幂等处理，事务消息不保证只投递一次

**适用场景**：电商下单后发送支付消息、订单完成后发送物流消息、账户变动后发送通知等

**优缺点**：
- 优点：解耦了业务与消息发送的强一致要求，无需本地消息表，性能好
- 缺点：依赖 RocketMQ 中间件，回查逻辑需要业务方实现，仅保证生产端原子性
